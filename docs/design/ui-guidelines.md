# UI 가이드라인 — 따뜻한 독서 노트 / 조용한 서재

> 이 문서는 이미 코드에 구현되어 있는 디자인 언어를 **사후 문서화**한 것입니다. 새 시각 언어를 제안하는 문서가 아니라, 화면을 새로 만들거나 고칠 때 "이미 있는 패턴 중 무엇을 재사용해야 하는가"를 빠르게 찾기 위한 참고서입니다.
> 캐릭터(시오리)의 조형 자체는 [`shiori-design-spec.md`](./shiori-design-spec.md)가 기준 문서이며, 이 문서는 그 캐릭터를 **화면에서 어떻게 배치하는가**만 다룹니다.
> 참고: `shiori-design-spec.md`의 5장(SVG/벡터 구현 명세)은 시오리가 손그림 SVG였던 시기의 기록입니다. 현재 `components/Shiori.tsx`는 사전 제작 PNG 에셋(`public/brand/shiori/shiori-<variant>.png`)을 사용합니다 — variant/size 체계와 사용 규칙은 여전히 유효하지만, "구현 명세"는 최신 구현과 다릅니다.

---

## 1. 제품 디자인 원칙

**한 문장 기준**: "조용한 서재에서 일본어 원문을 읽고, 모르는 단어를 책갈피처럼 모아 복습하는 앱." 새 화면이나 문구를 만들 때 이 문장에 비춰 판단합니다.

- **가벼운 원문 읽기 중심 앱** — 학습 대시보드가 아니라 리더 앱. 화면을 열었을 때 "지금 뭘 해야 하는지"가 숫자 나열보다 먼저 보여야 합니다.
- **따뜻한 독서 노트 톤** — 기능을 설명하는 말투가 아니라 노트에 적어두는 말투. 해요체가 기본이며, 정책/고지성 한 줄만 예외적으로 허용합니다(8장 참고).
- **조용한 서재** — 화면마다 강조점은 하나만. 같은 화면에 두 개의 "가장 중요한 것"이 동시에 있으면 안 됩니다.
- **색인 카드 / 책갈피 / 서가 메타포** — 리스트는 표(table)가 아니라 카드, 진행 상태는 책갈피, 컬렉션은 서가로 은유합니다.
- **핵심 루프를 항상 최우선으로** — 원문 읽기 → 모르는 단어 저장 → SRS 복습 → 다시 원문 읽기. 새 기능이 이 루프 진입/탈출을 어렵게 만들면 안 됩니다.
- **절제** — 화면 전체에서 "대담한 요소"는 하나만 허용합니다(히어로 일러스트, 시그니처 카피 등). 나머지는 조용하게. 필요 없는 장식은 넣지 않습니다.

## 2. 화면별 역할

| 화면 | 역할 한 줄 | 절대 되면 안 되는 것 |
|---|---|---|
| **책상** (`HomeDashboard.tsx`) | 서재 입구 — 오늘 무엇을 할지 3개 CTA + 오늘 요약 3칩 + 최근 단어 인덱스카드로 안내 | 통계 대시보드, 기능 목록 나열 |
| **분류** (`AnalyzeSection.tsx`) | 붙여넣은 원문을 빠르게 훑으며 단어 상태를 분류하는 보조 진입점 | 읽기 탭과 중복되는 별도의 "분석 도구" 느낌 |
| **읽기** (`ReadingTab.tsx`, `ReaderMode.tsx`) | 원문 리더 + 곁에 있는 메모장. 원문이 항상 화면의 주인공 | 후보 목록/저장 바구니가 원문보다 시각적으로 커지는 것 |
| **노트** (`VocabSection.tsx`) | 색인 카드함 — 저장한 단어를 다시 찾아보고 상태를 조정하는 곳 | raw 데이터 테이블, DB 관리자 화면 느낌 |
| **복습** (`StudySection.tsx`) | 책상 위 SRS 플래시카드 한 장. 카드 흐름 자체가 주인공 | 카드 앞에 통계 그리드가 먼저 보이는 것 |
| **덱** (`SharedDeckSection.tsx`) | 서가 — 책 표지 메타포로 추천/공유 덱을 훑어보고 담는 곳 | 수백~수천 단어를 필터 없이 한 번에 쏟아내는 것 |
| **통계/학습 일지** (`InfoSection.tsx`) | 오늘 학습을 되짚어보는 일지. 숫자가 아니라 "무슨 일이 있었는지" | 숫자 그리드만 있는 관리자 대시보드(그 버전은 `StatsPanel.tsx`로 이미 분리되어 복습 탭의 접힌 상세 안에만 존재) |

각 화면은 위 "역할 한 줄"을 벗어나는 새 기능이 생기면, 그 기능이 정말 이 화면에 있어야 하는지부터 재검토합니다.

## 3. 버튼 계층 규칙

| 계층 | 클래스 | 언제 쓰는가 |
|---|---|---|
| Primary | (클래스 없는 기본 `<button>`) | 화면당 정말 하나, 지금 하려는 핵심 행동("원문 읽기 시작", "저장", "제출") |
| Secondary | `secondary-button` | primary와 나란히 있는 대안 행동, 또는 primary가 없는 화면의 확정 행동("닫기", "새로고침", "이 덱 학습하기") |
| Ghost / subtle | `ghost-button` (+ `compact-button`) | 부가 행동, 토글, "더보기"류. 앱에서 가장 많이 쓰이는 계층 — 화면의 시각적 무게를 줄이고 싶을 때 기본으로 선택 |
| Destructive | `danger-button` 계열 | 아래 참고 |

**Destructive 버튼은 하나가 아니라 세 종류**이며, 서로 다른 의도를 인코딩합니다 — 무조건 통일하지 않습니다:
- `danger-button` — 일반적인 삭제 행동의 기본형.
- `danger-button-subtle` (`danger-button`과 함께 사용) — 실제로 삭제이지만, 같은 행 안의 다른 버튼(수정, 신고 등)과 시각적으로 경쟁하지 않아야 할 때. 아웃라인 칩으로 표현.
- `danger-secondary-button` — 되돌릴 수 있는 "취소"류 행동(예: 덱 공유 취소 — 실제 데이터는 지워지지 않음). 옅은 채움 칩으로 표현해 "진짜 삭제"보다 가볍게 보이도록 함.

새 삭제/취소 버튼을 추가할 때: **데이터가 실제로 사라지는가?** 사라지면 `danger-button`(+ 필요시 `danger-button-subtle`), 목록에서만 빠지고 원본은 남으면 `danger-secondary-button`.

역할별 특수 버튼(`success-button`/`warning-button` — 분류 화면의 상태 지정 버튼, `save-dock-primary-button` — 읽기 저장 바구니의 문맥 강조 버튼)은 그 화면 전용이며 다른 화면에 그대로 재사용하지 않습니다.

## 4. 카드 규칙

카드는 "종류"가 아니라 **조합 가능한 modifier**입니다. 베이스 클래스에 필요한 modifier를 얹어 씁니다.

- **베이스**: `panel-card` — 테두리 + 패딩 + 내부 grid gap을 주는 기본 카드 컨테이너.
- **modifier**:
  - `hero-card` — 더 큰 radius + 그림자. 화면의 대표 카드(히어로, 복습 카드)에 얹음.
  - `note-card` — 왼쪽 accent 보더의 메모/쪽지 느낌.
  - `card-stack-surface` — 뒤에 살짝 어긋난 종이가 쌓인 것처럼 보이는 의사 요소. 화면의 "주인공 카드"에만 사용 — 리스트 row마다 넣지 않음.
  - `paper-corner` — 종이 모서리가 접힌 듯한 디테일. 색인 카드/기록 카드류에 사용.
- **색인 카드 리스트**: `records-word-row`(InfoSection의 최근/자주 틀린 단어, VocabSection의 사용자 정의 용어 카드) — flex-wrap 기반이라 좁은 화면에서 가로 스크롤 없이 줄바꿈됨. 새 "단어 한 줄" 리스트를 만들 때 이 클래스를 우선 재사용.
- **필터 서랍**: `index-card-filter` — 점선 상단 테두리 + `memo-label`(작은 eyebrow 라벨)로 "카드함 필터를 당겨 여는" 느낌. 검색/필터 UI를 새로 만들 때 raw한 테두리 박스 대신 이 클래스를 재사용(노트 탭, 덱 탭 학습 목록에 이미 적용).
- **덱/책 표지**: `brand-deck-cover`(`BrandDeckCover` 컴포넌트) — JLPT 레벨/내가 공유함/공유 덱 3톤 체계. 새로운 덱 종류가 생겨도 이 3톤 체계 안에서 tone을 고르는 것을 우선 검토.

새 화면에 카드가 필요하면: 새 클래스를 만들기 전에 `panel-card` + 위 modifier 조합으로 표현되는지부터 확인합니다.

## 5. Empty / Loading / Success 상태 규칙

- **통일된 진입점**: `AppEmptyState`(`components/BrandElements.tsx`)가 유일한 empty-state 컴포넌트입니다. `mood`(시오리 variant) 또는 `icon`(정적 아이콘) + `title` + 선택적 `description` + children(액션 버튼) 순서 고정. 새 empty state를 만들 때 직접 마크업을 짜지 않고 이 컴포넌트를 사용합니다.
- **로딩**: 문장형 안내(`"~불러오는 중이에요..."`, `"~불러오는 중입니다"`) 또는 `AppEmptyState mood="loading"`. 스피너만 덩그러니 두지 않고 무엇을 기다리는지 한 줄로 씁니다.
- **성공**: `ShioriStamp variant="success"` — 완료 메시지 옆에 붙는 작은 도장. 항상 `sm` 크기 고정이며 별도의 "두 번째 큰 캐릭터"가 되지 않습니다.
- **톤**: empty/guide 문구는 해요체(`~없어요`, `~해보세요`)로 통일합니다. 정책/고지 한 줄만 예외로 허용(8장 참고). `<p className="empty">...`처럼 `AppEmptyState`를 거치지 않는 인라인 "결과 없음" 문구도 동일한 톤 규칙을 따릅니다.

## 6. 시오리(Shiori) 사용 규칙

전체 조형 스펙은 [`shiori-design-spec.md`](./shiori-design-spec.md) 참고. 여기서는 **배치 규칙**만 정리합니다.

- **화면당 큰 시오리(`size="lg"` 이상)는 최대 1개.** 히어로 위치 또는 empty-state 중 하나에만 등장하며, 같은 화면에 두 곳이 동시에 보이지 않게 조건부 렌더링을 유지합니다.
- **버튼 안에 시오리를 넣지 않습니다.** 버튼 옆에 별개 요소로 배치하는 것은 가능(예: 저장 바구니 토글 버튼 옆 `ShioriStamp`).
- **리스트 row마다 반복하지 않습니다.** 노트/덱/통계의 단어 리스트, 카드 그리드에는 시오리를 넣지 않습니다 — `VocabSection.tsx`의 단어 행에 시오리가 전혀 없는 것이 올바른 기준입니다.
- **컴포넌트별 용도**:
  - `ShioriCharacter` — 히어로/empty-state 등 화면급 일러스트.
  - `ShioriMark` — 제목 옆 작은 인라인 브랜드 마크(항상 `sm`).
  - `ShioriStamp` — "방금 완료됨" 도장(항상 `sm`, 라벨 선택).
  - `ShioriGuideCard` — 한 줄 힌트 카드(읽기 탭의 idle 안내 등). 액션 버튼이 있는 완전한 empty-state 대신 가벼운 힌트 한 줄이 필요할 때만 사용.
- **모달의 캐릭터 유무는 화면마다 선택 사항입니다.** `GlobalFeedbackModal`은 동반자 캐릭터가 있고 `MeaningFeedbackModal`은 없습니다 — 이는 규칙 위반이 아니라 "필요할 때만 추가"의 결과이므로, 통일을 위해 없는 곳에 새로 추가하지 않습니다.
- **새 시오리 이미지/포즈를 추가하지 않습니다.** 9종 variant(`default/hero/reading/classify/save/review/success/empty/loading`) 안에서 해결되는지 먼저 확인합니다.

## 7. 모바일 규칙

- **하단 탭 7개는 고정 구조입니다.** 탭 수를 줄이거나 재배치하는 것은 이 문서의 범위 밖이며, 별도 논의 없이 바꾸지 않습니다.
- **탭 아이템 최소 터치 높이 48px**(`--app-bottom-nav-item` `min-height:48px`)을 유지합니다. 폭이 좁아지는 것은 감수하되, 높이는 절대 줄이지 않습니다.
- **라벨은 2글자 내외로 짧게 유지합니다**(`mobileLabel`). 탭이 7개라 라벨이 길어지면 잘림/줄바꿈이 생깁니다.
- **`env(safe-area-inset-bottom)`을 항상 존중합니다** — 노치/홈 인디케이터가 있는 기기에서 하단 탭이 가려지지 않도록.
- **큰 리스트(구독 덱 단어 목록 등)는 고정 높이 스크롤 박스 + 검색/필터 + 점진적 로딩("더 보기")으로 제한합니다.** 페이지 전체 길이로 늘어나는 리스트를 모바일에 그대로 두지 않습니다.

## 8. 금지 표현

- **"공식 JLPT" 계열 표현 금지.** JLPT 관련 덱/단어는 항상 "추천 어휘", "학습 참고용 비공식 목록" 같은 표현만 사용합니다(`SharedDeckSection.tsx`의 기존 디스클레이머 문구가 기준).
- **"복사된 단어" 계열 표현 금지.** lexeme 기반 구독 덱은 실제로 개인 단어장에 복사되지 않으므로, "가져온 덱", "학습 목록에 추가" 같은 표현만 사용합니다.
- **원문 전체 저장/공유를 암시하는 문구 금지.** "원문 전체를 저장합니다", "원문 전체를 공유합니다" 같은 표현은 절대 쓰지 않습니다. 항상 부정형("원문 전체는 저장하지 않아요")으로만 언급합니다.
- **정책 문구의 톤은 해요체로 통일합니다.** ("~하지 않아요", "~돼요") 같은 문장 안에서 해요체와 습니다체가 섞이지 않게 합니다(이번 정리에서 `InfoSection.tsx`/`SharedDeckSection.tsx`/`AnalyzeSection.tsx`의 저장 정책 문구를 통일했습니다).
- **내부 상태값(영문)을 사용자 문구에 그대로 노출하지 않습니다.** `unknown`/`uncertain`/`unclassified` 같은 코드 값 대신 "모르는 단어", "헷갈리는 단어", "분류하지 않은 단어" 같은 한국어 사용자 문구로 항상 번역합니다.

## 9. 새 화면 추가 시 체크리스트

새 화면/기능을 추가하기 전에 아래를 확인합니다.

1. **이 화면은 핵심 루프(읽기→저장→복습) 중 어디에 있는가?** 루프 밖의 보조 화면이라면 진입/탈출이 1~2단계 이내인지 확인합니다.
2. **이 화면의 "역할 한 줄"을 정의했는가?** (2장 표 형식으로) 정의하지 못하면 화면을 쪼개거나 기존 화면에 통합할 수 있는지 재검토합니다.
3. **버튼은 primary 1개 이하로 시작하는가?** 강조 버튼이 2개 이상이면 우선순위를 다시 정합니다.
4. **카드/리스트는 기존 클래스(`panel-card`+modifier, `records-word-row`, `index-card-filter`)로 표현되는가?** 새 CSS 클래스를 만들기 전에 기존 조합을 먼저 시도합니다.
5. **empty/loading/success 상태를 `AppEmptyState`/`ShioriStamp`로 처리했는가?**
6. **시오리를 추가한다면**: 화면에 이미 큰 시오리가 있는가? 버튼 안에 넣지 않았는가? row마다 반복하지 않았는가?
